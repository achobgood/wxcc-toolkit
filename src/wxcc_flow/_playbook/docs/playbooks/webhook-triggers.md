# Webhook Triggers Playbook

## Overview

This playbook covers how to trigger Webex Connect flows from external systems via inbound webhooks. Use this pattern when a database, application, or third-party service needs to kick off a Connect flow (e.g., order-ready notification, status change alert, data sync).

**Key distinction from AI Agent flows:** Webhook-triggered flows use the **Start node** with an Inbound Webhook trigger, NOT the Receive node / AI Agent Event pattern. The Receive node is a mid-flow pause for already-running flows — the Start node is the flow entry point.

---

## 1. Start Node Webhook Configuration

1. Create a new flow (or open an existing flow's Start node)
2. In the Start node trigger configuration, select **Webhook**
3. Either select an existing webhook or **create a new webhook event** inline
4. Enter a name for the webhook
5. In the **PROVIDE SAMPLE INPUT** area, paste a sample JSON payload:
   ```json
   {
     "order_id": "ORD-4521",
     "customer_phone": "+15551234567",
     "location_name": "Main Street Store"
   }
   ```
6. Click **PARSE** — Connect extracts field names and makes them available as flow variables
7. Save the configuration

Once saved, Connect auto-generates a unique webhook URL. The flow must be **published (made live)** for the webhook to accept requests.

---

The URL is **auto-generated by Connect** — you cannot choose or construct it.

## 3. Payload Parsing

- Paste a **sample JSON or XML payload** into the **PROVIDE SAMPLE INPUT** area, then click **PARSE**
- Connect uses **JPath** (JSON) or **XPath** (XML) to extract fields
- Nested fields use dot notation: `data.attributes.title` for `{"data": {"attributes": {"title": "value"}}}`
- After parsing, all extracted variables become available via the variable picker throughout the flow

### Variable Reference Format

```
$(n1.inboundWebhook.fieldName)
```

Where `n1` is the Start node number and `fieldName` matches a key from the parsed sample. The general variable pattern `$(n[nodeID].[nodeType].[fieldName])` is confirmed in official docs; `inboundWebhook` as the specific node type identifier for Start node webhook fields is consistent with platform behavior but not explicitly confirmed in official documentation.

**Important:** Only fields present in the parsed sample input will be available as named variables. Fields sent but not in the sample are received but harder to reference. Update the sample input if your payload structure changes.

## 4. Authentication & Security

Authentication is **optional but recommended** for production. Three layers available:

### Service Key

Pass the service key in the request header:
```
key: {your_service_key}
```

Found in: Service dashboard > API tab (masked by default; click eye icon to reveal).

### JWT Token

Generate a JWT signed with HMACSHA256 using your Base64-encoded Service Secret:
- Payload requires `iss` (Service ID as issuer), `iat` (issued-at timestamp)
- Optional `exp` (expiry) — tokens expire after 60 minutes
- Header: `key: {jwt_token}`
- If both service key and JWT are passed, JWT takes priority

### Signature Validation (HMAC)

- Optional signature validation on the request body (SHA-256 or SHA-512 per API reference; specific algorithms not listed in help pages)
- Invalid signatures return HTTP 400 (error code 7029: Mandatory Signature Missing; 7028: Invalid signature)
- Configured on the webhook definition, independent of service key / JWT

### IP Allowlisting

IP allowlisting exists as a feature (error code 7010: Source IP not allow listed confirms it), but configuration steps are not documented in official help pages.

## 5. HTTP Method & Content Types

| Property | Value |
|----------|-------|
| **HTTP Method** | **POST only** (consistent with platform behavior; not stated verbatim in official docs) |
| **Content Types** | `application/json` and `application/xml` |
| **Payload Size Limit** | Default 256 KB (configurable at tenant level). Exceeding returns error 430. |

## 6. Testing Webhooks

### curl Example

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  -H "key: {your_service_key}" \
  -d '{"order_id": "ORD-4521", "customer_phone": "+15551234567", "location_name": "Main Street Store"}' \
  "https://{your-webhook-url}"
```

### Response Behavior

The webhook is **asynchronous**. The HTTP response confirms the request was queued — it does NOT return the flow's output.

> **Needs verification:** This async behavior applies to standard inbound webhooks. The AI Agent Event trigger (Virtual Agent V2 → Flow Outcomes) IS synchronous. There may be other sync mechanisms for Connect ↔ Flow Designer data exchange not yet documented here. If you need a Connect flow to return data synchronously to Flow Designer, verify whether an alternative trigger type or integration pattern supports it before assuming the webhook is the only option.

| Response Code | Meaning |
|---------------|---------|
| 1002 | Queued — request accepted |
| 7000 | Invalid JSON |
| 7001 | Authentication failed |
| 7002 | Service Key missing |
| 7005 | Internal error |
| 7007 | Service inactive |
| 7008 | Invalid or unsaved event endpoint |
| 7020 | Max transaction limit reached |
| 7027 | Invalid XML |
| 7028 | Invalid signature |
| 7029 | Mandatory Signature Missing |
| 430 | Payload too large |

### Verifying Flow Execution

- **Flow Debug** panel in the flow builder — shows the 10 most recent executions with step-by-step traces
- **Transaction logs** — available for 30 days, appear within ~2 minutes of execution

## 7. Flow Structure (Webhook-Triggered)

Since this is NOT an AI Agent flow, the AI agent constraints do not apply:

- **No 30-second timeout** (standard flow timing applies)
- **No prohibited nodes** — you CAN use Delay, Receive (mid-flow), Call Workflow, Social Hour
- **No Flow Outcomes node needed** — there is no AI agent to return data to

### Example: Order Notification

```
Start (Webhook: order_id, customer_phone)
  → HTTP Request (GET order details by order_id)
  → Call User (dial customer_phone)
    → onAnswer → Voice Node Group [ Play (TTS) → End ]
    → onBusy → End
    → onNoAnswer → End
```

### Example: SMS Alert

```
Start (Webhook: phone, message)
  → SMS (send $(n1.inboundWebhook.message) to $(n1.inboundWebhook.phone))
  → End
```

### Example: Data Sync

```
Start (Webhook: payload)
  → Evaluate (transform data)
  → HTTP Request (POST to external system)
  → End
```

## 8. One Trigger Per Flow

- A flow has **one Start node** with **one webhook trigger** configuration
- If you need different webhooks triggering different logic, create separate flows
- The Start node supports **conditional triggers** using AND/OR operators with Equals, Not equals, Contains, Contains ignore-case, In, Not in, Starts with, Ends with, Greater than, Less than, Greater than equal to, Less than equal to, Between, Equals ignore-case, and Regex to filter which payloads actually execute the flow

---

## 9. Webhook vs Custom Event vs AI Agent Event

| Feature | Inbound Webhook | Custom Event (v1) | AI Agent Event |
|---------|----------------|-------------------|----------------|
| **Configured in** | Start node | Start node | Start node (AI Agent integration) |
| **Payload** | Any JSON/XML | Pre-defined parameters only | Slot entities from AI Agent Studio |
| **Auth** | Optional (key, JWT, signature) | Mandatory (service key) | Automatic (Webex CI) |
| **Response** | Async (queue confirmation) | Async (queue confirmation) | Sync (Flow Outcomes returned to agent) |
| **Use case** | External system integrations | Structured business events | AI agent action fulfillment |

For external system integrations (database triggers, third-party webhooks), use **Inbound Webhook**.

---

## 10. Triggering from a Database

Databases don't natively fire webhooks. You need a middleware layer:

| Approach | How |
|----------|-----|
| **Application-level trigger** | Your app updates the DB, then POSTs to the Connect webhook in the same transaction/handler |
| **Database trigger + function** | Postgres: use `pg_net` extension or a trigger that calls a serverless function. Supabase: use Database Webhooks or Edge Functions. |
| **Change Data Capture (CDC)** | Tools like Debezium stream DB changes to Kafka/webhooks |
| **Polling service** | Lightweight service polls the DB for status changes and fires webhooks |

The simplest path: have your application code POST to the Connect webhook URL immediately after updating the order status in the database.

## 11. Known Gotchas

| Issue | Fix |
|-------|-----|
| Webhook returns 7001 (auth failed) | Service key missing or malformed in `key` header |
| Variables show as empty in downstream nodes | Sample input not parsed — go back to Start node and click Parse |
| Flow doesn't trigger on webhook POST | Flow not published (made live) |
| Webhook URL not visible | Save the webhook configuration first, then double-click Start node to view |
| Payload rejected (error 430) | Payload exceeds 256 KB default limit |
| Nested fields not accessible | Use dot notation in sample input; re-parse after updating |
| Webhook fires but flow errors on first node | Payload field names don't match sample input keys (case-sensitive) |
